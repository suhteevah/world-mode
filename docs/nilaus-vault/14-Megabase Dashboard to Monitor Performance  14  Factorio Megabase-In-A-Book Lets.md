## Summary
### Key Lessons
- Build a global monitoring dashboard using red/green wire networks already embedded in city block infrastructure
- Monitor both absolute supply (loads available) AND net supply-demand difference - you need both views
- Supply-demand dashboard: positive = surplus (green/good), negative = deficit (red/bad), zero = balanced
- Use arithmetic combinators to convert station signals to item-specific signals on the global network
- Add monitoring to every new production build as you create it, not as a bulk task afterwards (like adding tests in software development)

### Design Principles
- Green wire = supply side signals, Red wire = demand side signals across the global network
- Each station outputs its supply/demand count as a generic signal (L), then an arithmetic combinator converts it to the specific item signal
- Dashboard uses binary lamp encoding: 6 lamps represent values -1, -2, -4, -8, -16, -32 for visual net supply-demand display
- Multiply demand by -1 then add to supply to get net difference on a single combinator
- Keep dashboard signals isolated - prevent signal feedback loops back into the train network

### Ratios & Numbers
- Dashboard binary display: 6 lamps covering range of -63 to +63 in supply-demand difference
- Supply indicator: number of full train loads available at provider stations
- Demand indicator: number of train loads being requested by consumer stations
- Positive net value = trains waiting to be picked up; negative = stations starving for deliveries

### Mistakes to Avoid
- Signal feedback loops: connecting dashboard combinators back to the station network corrupts train limit signals
- Testing with value -15 appeared broken because it summed to exactly zero (no lamps lit) - edge case, not a bug
- Do not rely on supply-only monitoring - a station can show surplus while downstream consumers are starving due to transit delays
- Forgetting to multiply demand side by -1 before combining with supply gives incorrect net values
